## 行为准则

### 我们的承诺

为了营造一个开放和友好的环境，我们作为贡献者和维护者承诺：无论年龄、体型、残疾、种族、性别认同和表达、经验水平、教育程度、社会经济地位、国籍、外貌、种族、宗教或性取向如何，参与我们的项目和社区都将为每个人提供无骚扰的体验。

### 我们的标准

**积极行为示例：**

- 使用友好和包容的语言
- 尊重不同的观点和经验
- 优雅地接受建设性批评
- 关注对社区最有利的事情
- 对其他社区成员表示同理心

**不可接受的行为示例：**

- 使用性化的语言或图像，以及不受欢迎的性关注或性骚扰
- 捣乱、侮辱/贬损评论以及人身或政治攻击
- 公开或私下骚扰
- 未经明确许可，发布他人的私人信息
- 其他在专业环境中可能被合理认为不适当的行为

## 如何贡献

### 报告 Bug

如果您发现了 bug，请创建一个 [Issue](https://github.com/YYC-Cube/yyc3-claw/issues) 并包含：

1. **清晰的标题**：简洁描述问题
2. **复现步骤**：详细的复现步骤
3. **预期行为**：您期望发生什么
4. **实际行为**：实际发生了什么
5. **环境信息**：
   - 操作系统
   - Node.js 版本
   - 项目版本
6. **截图**：如果适用，添加截图帮助解释问题

**Bug 报告模板：**

```markdown
## Bug 描述
[清晰简洁地描述 bug]

## 复现步骤
1. 执行 '...'
2. 点击 '....'
3. 滚动到 '....'
4. 看到错误

## 预期行为
[描述您期望发生的情况]

## 实际行为
[描述实际发生的情况]

## 截图
[如果适用，添加截图]

## 环境信息
- 操作系统: [例如 macOS 14.0]
- Node.js 版本: [例如 18.0.0]
- 项目版本: [例如 1.0.0]

## 附加信息
[添加任何其他关于问题的信息]
```

### 建议新功能

我们欢迎新功能建议！请创建一个 [Issue](https://github.com/YYC-Cube/yyc3-claw/issues) 并包含：

1. **功能描述**：清晰描述您希望添加的功能
2. **使用场景**：描述这个功能解决什么问题
3. **建议方案**：如果您有想法，描述可能的实现方式
4. **替代方案**：描述您考虑过的替代方案

**功能请求模板：**

```markdown
## 功能描述
[清晰简洁地描述您希望添加的功能]

## 使用场景
作为 [用户类型]，我希望 [执行某操作]，以便 [达到某目的]

## 建议方案
[描述您建议的解决方案]

## 替代方案
[描述您考虑过的替代解决方案]

## 附加信息
[添加任何其他关于功能请求的信息]
```

### 改进文档

文档改进包括：

- 修正拼写或语法错误
- 添加缺失的文档
- 改进现有文档的清晰度
- 翻译文档

## 开发设置

### 1. Fork 项目

点击项目页面右上角的 "Fork" 按钮。

### 2. 克隆仓库

```bash
git clone https://github.com/YOUR_USERNAME/yyc3-claw.git
cd yyc3-claw
```

### 3. 添加上游仓库

```bash
git remote add upstream https://github.com/YYC-Cube/yyc3-claw.git
```

### 4. 安装依赖

```bash
pnpm install
```

### 5. 创建分支

```bash
git checkout -b feature/your-feature-name
```

### 6. 开发

```bash
# 启动开发模式
pnpm dev

# 运行测试
pnpm test:watch

# 代码检查
pnpm lint
```

## 提交规范

我们使用 [Conventional Commits](https://www.conventionalcommits.org/) 规范。

### 提交格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Type 类型

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat: 添加用户认证功能` |
| `fix` | Bug 修复 | `fix: 修复登录验证错误` |
| `docs` | 文档更新 | `docs: 更新 API 文档` |
| `style` | 代码格式（不影响代码运行） | `style: 格式化代码` |
| `refactor` | 重构（既不是新功能也不是修复 bug） | `refactor: 重构用户服务` |
| `perf` | 性能优化 | `perf: 优化查询性能` |
| `test` | 添加测试 | `test: 添加认证模块测试` |
| `chore` | 构建过程或辅助工具的变动 | `chore: 更新依赖` |
| `revert` | 回滚 | `revert: 回滚登录功能` |

### Scope 范围（可选）

- `auth` - 认证模块
- `ai-family` - AI Family 模块
- `skills` - 技能系统
- `mcp` - MCP 协议
- `multimodal` - 多模态处理
- `core` - 核心模块

### Subject 主题

- 使用祈使语气，现在时态："change" 而不是 "changed" 或 "changes"
- 第一个字母小写
- 结尾不加句号

### 示例

```bash
# 新功能
git commit -m "feat(auth): 添加 OAuth2.0 认证支持"

# Bug 修复
git commit -m "fix(ai-family): 修复智能体任务调度问题"

# 文档更新
git commit -m "docs: 更新开发者指南"

# 多行提交
git commit -m "feat(skills): 添加代码审查技能

- 实现代码质量分析
- 添加安全漏洞检测
- 支持多种编程语言

Closes #123"
```

## Pull Request 流程

### 1. 确保代码质量

```bash
# 运行测试
pnpm test

# 类型检查
pnpm typecheck

# 代码检查
pnpm lint

# 构建项目
pnpm build
```

### 2. 更新分支

```bash
git fetch upstream
git rebase upstream/main
```

### 3. 推送分支

```bash
git push origin feature/your-feature-name
```

### 4. 创建 Pull Request

1. 访问您的 fork 页面
2. 点击 "New Pull Request"
3. 填写 PR 模板

**Pull Request 模板：**

```markdown
## 变更类型
- [ ] Bug 修复
- [ ] 新功能
- [ ] 重构
- [ ] 文档更新
- [ ] 其他

## 变更描述
[清晰描述您的变更]

## 相关 Issue
Closes #

## 测试
- [ ] 已添加测试
- [ ] 所有测试通过

## 检查清单
- [ ] 代码遵循项目代码规范
- [ ] 已进行自我审查
- [ ] 已更新相关文档
- [ ] 变更不会产生新的警告
- [ ] 已添加必要的测试
- [ ] 所有测试通过
```

### 5. 等待审查

- 维护者会审查您的 PR
- 根据反馈进行必要的修改
- 通过审查后会被合并

## 代码审查标准

### 代码质量

- ✅ 代码清晰易读
- ✅ 遵循项目代码规范
- ✅ 没有重复代码
- ✅ 适当的注释
- ✅ 良好的命名

### 测试覆盖

- ✅ 单元测试覆盖新代码
- ✅ 边界情况测试
- ✅ 错误处理测试
- ✅ 所有测试通过

### 文档

- ✅ 更新相关文档
- ✅ API 文档更新
- ✅ README 更新（如需要）

### 性能

- ✅ 没有性能退化
- ✅ 合理的资源使用
- ✅ 没有内存泄漏
