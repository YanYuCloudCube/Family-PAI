### 成员详细定义

#### 🧭 言启·千行（Navigator）

| 属性 | 定义 |
|------|------|
| **角色定位** | 系统的"耳朵"与"翻译官"，用户意图进入YYC³世界的第一道门户 |
| **推理类型** | 规划推理、资源调度 |
| **核心能力** | 自然语言理解、意图识别、上下文管理 |

**核心职责**：
1. 自然语言理解（NLU）：解析用户查询
2. 意图识别与路由：判断意图类型并路由给专业成员
3. 上下文管理：维护对话上下文，实现多轮交互连贯性

**文件架构**：
```

│   ├── context-manager.service.ts     # 对话上下文管理
│   └── intent-classifier.service.ts   # 意图分类器
├── nlq/
│   ├── query-parser.service.ts        # 自然语言查询解析
│   └── query-to-api.mapper.ts         # 查询映射为API调用
└── prompts/
    ├── intent-detection.prompt.ts     # 意图识别提示词
    └── query-parsing.prompt.ts        # 查询解析提示词
```

| 第八层 | 应用表现层 | 数据可视化与报表生成 | ECharts, D3.js |
| 第七层 | 业务逻辑层 | 业务流程编排与规则处理 | Node.js, Python |
| 第六层 | AI智能层 | AI模型推理与知识管理 | TensorFlow, PyTorch |

## 标准规范体系

### 开发规范

| 规范类别 | 具体要求 |
|----------|----------|
| **命名规范** | 文件名使用kebab-case，类名使用PascalCase，变量名使用camelCase |
| **代码规范** | 遵循ESLint/Prettier配置，统一代码风格 |
| **注释规范** | 必须包含文件头注释，复杂逻辑必须添加行内注释 |
| **提交规范** | 使用Conventional Commits格式，关联Issue编号 |

### 文档规范

| 规范类别 | 具体要求 |
|----------|----------|
| **元数据规范** | 必须包含@file、@description、@author、@version等字段 |
| **格式规范** | 使用Markdown格式，统一标题层级 |
| **内容规范** | 包含概述、详细说明、示例、参考链接 |
| **版本规范** | 遵循语义化版本控制，记录变更历史 |

### 安全规范

| 规范类别 | 具体要求 |
|----------|----------|
| **认证规范** | 使用JWT Token认证，支持OAuth2.0 |
| **授权规范** | 基于RBAC的权限控制，最小权限原则 |
| **数据规范** | 敏感数据加密存储，传输使用HTTPS |
| **审计规范** | 记录所有操作日志，支持审计追踪 |
